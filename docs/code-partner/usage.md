## Skill 工作原理

`code-partner` 是一个 Claude Code Skill，适用于存在实际代码变更、且用户希望先分析、评估、权衡再实施的开发任务。在执行前，它会先进行技术益害评估。

对于纯查询、只读定位、简单文档修改、显式安全操作、代码解释、日志处理或其他低风险轻量任务，不应强行套用完整流程。

### 触发条件

以下情况应优先考虑使用 skill：
- 用户明确要求先评估收益、风险、替代方案或技术取舍，再决定是否实施
- 用户要求先拆分子任务、梳理改动边界，再进入实现
- 高影响代码决策：重构、架构调整、性能方案、并发/一致性问题、第三方依赖引入、PR 改动方向审查
- 其他明显需要“先判断、后编码”的代码任务

以下情况通常不需要使用完整流程：
- 用户已明确实现方向，并要求直接修改、直接修 Bug、直接补测试
- 普通编码实现，即使涉及多文件改动，只要不需要前置权衡
- 纯查询或只读定位
- 简单文档修改
- 显式安全操作
- 代码解释
- 日志处理
- 常规重命名、机械性修改等低判断成本任务

### 评估流程

```
1. 识别任务类型（需求/优化/Bug修复）
2. 进行益处检查
3. 进行风险评估
4. 判定风险等级（🟢低 / 🟡中 / 🔴高）
5. 根据等级决定响应策略
```

## 完整工作流程

### 场景 1：低风险变更（🟢，轻量评估模式）

```
用户：优化这个函数的性能

Claude：
✅ 技术益害评估：
  益处：提升查询效率 50%
  风险：🟢 低风险 - 局部优化，易回滚

→ 直接执行变更
→ 记录优化日志（如有实际代码变更）
→ 补充必要测试
→ 验证核心功能
```

### 场景 2：中风险变更（🟡，轻量评估模式）

```
用户：引入 Redis 缓存

Claude：
⚠️ 技术益害评估：
  益处：显著降低数据库负载
  风险：🟡 中风险
    - 增加系统复杂度
    - 需要额外维护缓存服务
    - 可能出现数据不一致

💡 建议：
  1. 先评估是否有更简单的方案（如内存缓存）
  2. 确认是否真的需要分布式缓存
  3. 考虑使用方案 2 减少依赖

→ 等待用户确认
→ 执行（或调整方案）
→ 记录优化日志（如该方案最终落地）
...
```

### 场景 3：高风险变更（🔴，轻量评估模式）

```
用户：把所有单例改成全局变量

Claude：
🚨 技术益害评估：
  风险：🔴 高风险
    - 破坏单例模式的线程安全性
    - 可能导致状态不一致
    - 影响整个系统稳定性

❌ 强烈不建议此方案

✅ 推荐替代方案：
  1. 保持单例模式，通过依赖注入解决
  2. 使用上下文管理器控制生命周期
  3. 考虑使用工厂模式

请选择替代方案或提供更多背景信息...
```

## 响应模式

### 1. 轻量评估模式（默认）

适用于大多数开发任务的前置判断、方案讨论和小中型改动。此时应优先给出简洁结论，而不是一上来输出整块日志模板。

推荐结构：

```markdown
技术益害评估：
- 益处：...
- 风险：🟢/🟡/🔴 ...
- 决策：直接执行 / 先确认 / 提供替代方案
```

如需继续执行，可补充：
- 计划如何实施
- 是否需要补测试
- 是否建议记录日志

### 2. 完整日志模式（按需使用）

适用于以下情况：
- 已完成重要代码变更
- 需要正式沉淀优化记录
- 任务风险较高、影响面较大、后续需要追溯

这时再输出完整日志条目，并写入 `project_logs/optimization/YYYY-MM-DD.md`。

### 3. 选择原则

- **先轻后重**：先帮助用户判断值不值得做、风险大不大，再决定是否进入完整日志模式
- **有记录需求再展开**：只有在真正发生重要代码变更时，才输出完整日志条目
- **不要机械套模板**：查询、解释、只读定位或轻量讨论场景，保持简洁即可

## 手工日志记录规范

### 基本做法

每次完成**重要代码变更**后，直接将日志条目写入：

- `project_logs/optimization/YYYY-MM-DD.md`

推荐流程：
1. 检查当日日志文件是否存在
2. 不存在则创建，并写入日期标题
3. 按模板追加新的“优化日志条目”

### 推荐模板

```markdown
# 优化日志 - 2026-03-11

### 【优化日志条目】

**变更类型**：优化
**时间戳**：2026-03-11 10:30:00
**负责人**：Claude

#### 变更目的
优化用户查询接口性能，解决 N+1 问题

#### 变更内容
- **具体修改**：使用 `selectinload()` 预加载关联数据
- **涉及模块**：`user_service`, `database_layer`
- **实施方法**：SQLAlchemy ORM 批量加载

#### 验证标准
- **成功指标**：API 响应时间降至 150ms 以下
- **测试方案**：单元测试 + 性能测试
- **风险评估**：中风险，需关注预加载带来的内存占用

#### 后续建议
- **相关影响**：其他列表查询可参考此方案同步优化
- **待办事项**：审查其他循环查询代码

---
✅ 变更已记录至 `project_logs/optimization/2026-03-11.md`
```

## 常见使用场景

### 1. 新功能开发

**场景**：添加用户认证功能

记录到日志时，应包含：
- 变更类型：需求
- 变更目的：实现 JWT 用户认证
- 涉及模块：`auth`, `middleware`
- 具体修改：添加 JWT 签发和验证逻辑
- 实施方法：使用 PyJWT 库实现
- 成功指标：成功签发 token，正确验证过期 token
- 测试方案：单元测试 + 集成测试
- 风险评估：中风险，需保证密钥安全
- 相关影响：需要更新 API 文档
- 待办事项：配置密钥轮换策略、添加 refresh token 机制

### 2. 性能优化

**场景**：优化慢查询

记录到日志时，应包含：
- 变更类型：优化
- 变更目的：优化搜索接口性能
- 涉及模块：`search_service`
- 具体修改：添加复合索引，优化查询条件
- 实施方法：数据库索引优化
- 成功指标：查询时间 < 200ms
- 测试方案：使用生产数据集压测
- 风险评估：低风险，索引会增加写入开销
- 相关影响：需要定期监控索引大小

### 3. Bug 修复

**场景**：修复并发问题

记录到日志时，应包含：
- 变更类型：Bug修复
- 变更目的：修复并发写入数据竞争
- 涉及模块：`data_writer`
- 具体修改：添加线程锁保护共享状态
- 实施方法：使用 `threading.Lock()`
- 成功指标：无数据丢失、无死锁发生
- 测试方案：并发压力测试
- 风险评估：中风险，锁可能影响性能
- 相关影响：需监控锁等待时间

## 最佳实践

### 1. 及时记录

- **重要代码变更完成后尽快记录**，不要拖延
- 多个紧密相关的小改动可合并成一个条目

### 2. 详略得当

- **高风险变更**：详细记录所有字段
- **低风险变更**：可简化非关键字段

### 3. 日志组织

- 以日期为单位存放日志文件，例如 `project_logs/optimization/2026-03-11.md`
- 建议一个变更对应一个完整条目
- 重大变更可在条目内增加子项说明

### 4. 定期回顾

- 每周回顾优化日志
- 识别常见问题和模式
- 持续改进开发流程

### 5. 团队协作

- 在 `.gitignore` 中排除 `project_logs/`（如果是敏感信息）
- 或者将日志纳入版本控制（如果适合团队分享）
- 统一使用日志模板，保持格式一致

## 常见问题

### Q1: 日志文件太大会影响性能吗？

**A**: 不会。日志按天分割，每天一个文件，单个文件通常只有几十 KB。

### Q2: 可以手动编辑日志文件吗？

**A**: 可以，且这是推荐方式。重点是保持模板一致，避免遗漏“变更目的、变更内容、验证标准、后续建议”等关键信息。

### Q3: 日志没有版本号怎么办？

**A**: 现在不再依赖自动版本号。按日期文件 + 条目结构已经足够满足追溯需求；如果团队需要，也可以在条目标题中手工补充内部编号。

### Q4: 如何迁移旧日志？

**A**: 将旧的日志文件复制到 `project_logs/optimization/` 目录，文件名格式为 `YYYY-MM-DD.md`。
